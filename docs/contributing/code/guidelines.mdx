# 일반 지침

모든 유형의 기여를 위해 염두에 두어야 할 몇 가지 사항은 다음과 같습니다:

- ["포크 및 풀 리퀘스트"](https://docs.github.com/en/get-started/exploring-projects-on-github/contributing-to-a-project) 워크플로를 따르세요.
- 풀 리퀘스트를 열 때 체크된 풀 리퀘스트 템플릿을 작성하세요. 관련 문제를 언급하고 관련 유지 관리자를 태그하세요.
- 리뷰 요청 전에 PR이 포맷팅, 린트 검사 및 테스트 검사를 통과하는지 확인하세요.
  - 현재 진행 상황에 대한 댓글이나 피드백을 원하시면, 이슈나 토론을 열고 유지 관리자를 태그하세요.
  - 이러한 검사를 로컬에서 실행하는 방법은 [테스트](/docs/contributing/code/setup#testing) 및 [포맷팅 및 린트](/docs/contributing/code/setup#formatting-and-linting) 섹션을 참조하세요.
- 하위 호환성은 중요합니다. 중요한 버그 및 보안 수정이 아닌 한, 귀하의 변경 사항은 파괴적이지 않아야 합니다.
- 새로운 PR을 열기 전에 이미 열린 중복 PR이나 이슈를 찾아보세요.
- 범위는 가능한 한 격리된 상태로 유지하세요. 일반적으로 귀하의 변경 사항은 한 번에 하나의 패키지에만 영향을 미쳐야 합니다.

## 버그 수정

우리는 버그 수정을 장려하고 감사하게 생각합니다. 우리는 다음을 요청합니다:

- 유지 관리자가 버그를 재현할 수 있도록 충분한 세부정보로 버그를 설명하세요.
  - 동반되는 이슈가 있는 경우, 이를 링크하세요. PR이 병합될 때 이슈가 자동으로 닫히도록 `Fixes`로 접두사를 붙이세요.
- 가능하면 파괴적 변경을 피하세요.
- 버그 수정이 없으면 실패하는 단위 테스트를 포함하세요.

버그를 발견했지만 수정 방법을 모르는 경우, 버그를 발견한 환경을 자세히 설명하는 이슈를 열어주시기 바랍니다.

## 새로운 기능

우리는 새로운 기능에 대해 높은 기준을 유지하고자 합니다. 일반적으로 기존 GitHub 토론이나 이슈 없이 외부 기여자로부터 새로운 핵심 추상화, 인프라 변경, 종속성 변경 또는 새로운 에이전트/체인을 수용하지 않습니다.

- 새로운 기능은 문서, 단위 테스트 및 (적절한 경우) 통합 테스트와 함께 제공되어야 합니다.
- 새로운 통합은 문서, 단위 테스트 및 (적절한 경우) 통합 테스트와 함께 제공되어야 합니다.
  - 새로운 통합 기여에 대한 자세한 내용은 [이 페이지](/docs/contributing/integrations)를 참조하세요.
- 새로운 기능은 더 이상 사용되지 않는 메서드나 클래스를 상속하거나 사용해서는 안 됩니다.
- 보안 취약점이나 보고서로 이어질 가능성이 있는 기능은 거부할 것입니다.
- 하드 종속성을 추가하지 마세요. 통합은 선택적 종속성을 추가할 수 있습니다.
